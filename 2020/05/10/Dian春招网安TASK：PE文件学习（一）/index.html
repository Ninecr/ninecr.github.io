PE文件简介

PE文件的全称是Portable Executable，意为可移植的可执行的文件，常见的EXE、DLL、OCX、SYS、COM都是PE文件，PE文件是Windows操作系统上的程序文件（可能是间接被执行，如DLL）。此处需要知道的是，尽管DLL文件不能直接执行，但它与EXE文件使用完全相同的PE格式，仅用一个字段标识出这个文件是EXE还是DLL。
PE文件有一个共同特点：前两个字节为4D 5A（MZ）。如果一个文件前两个字节不是4D 5A则其肯定不是可执行文件。
PE文件是跨win32平台的，只要运行在Windows上，不管是在什么体系的CPU上都可以运行。

PE文件基本概念

  • PE文件使用平面的地址空间，所有代码和数据都合并在一起，组成一个很大的结构
  • 文件的内容被分为不同的区块，块中包含代码和数据，每个区块在内存中都有其对应的权限，比如有的块只读，有的只写或者有的只可执行
  • 每个区块在内存中按页边界对齐，区块没有大小限制，是一个连续的结构，每个块都有对应的属性
  • PE文件不是作为单一内存映射文件被装入内存的，但在内存中的数据结构布局与在磁盘中是一致的

PE文件结构

  • DOS头

  • PE头

  • 块/节表

  • 块/节(section)：

    • .data：初始化的数据块，也就是全局变量
    • .rdata：是运行时的只读数据，也就是所说的const常量
    • .text：是在编译或者汇编结束时产生的一种区块，是指令的代码
    • .idata：包含其他外来dll的函数以及数据信息，即输入表
    • .rsrc：包含全部的资源，如图标、菜单、位图等等
    • …

  • 相关名词

    • 基地址(ImageBase)：内存中PE文件的头地址，也是这个模块（PE文件）的句柄，可用函数HMODULE GetModuleHandle(); 获取
    • 文件的偏移地址：PE文件中各个部分相对于文件头的偏移
    • 虚拟地址(VirtuallAddress VA)：应用程序访问的逻辑地址也就是它的虚拟地址
    • 相对虚拟地址(RVA)：PE结构被映射到内存中后，某个位置所在内存相对于基地址的偏移

1.DOS头

PE文件的第一个字节起始于一个传统的MS-DOS头，被称为IMAGE_DOS_HEADER，这个结构体完整的定义如下：

  • e_magic：结构体的第一个成员2个字节是可执行文件的标志信息
  • e_lfanew：结构体最后一个成员，指向PE文件头的指针，这个在偏移3c处
  • 注：DOS头和PE头中间的空余位置是一些垃圾值以及编译器填充的一些“is program cannot be run in DOS mode.”或“This program must be run under Win32”等信息，以及空的0补满一页

2.PE头

PE相关结构NT映像头(IMAGE_NT_HEADER)：

  • NT头的第一个成员是”PE\0\0”（0X50 0X45 0X00 0X00四字节的签名），标志PE文件头的开始，验证其为有效PE文件

  • 第二个成员是标准PE头(_IMAGE_FILE_HEADER)：

    • Machine：记载可执行文件的目标CPU类型

      ValueMeaning
      IMAGE_FILE_MACHINE_I386 0x014cx86
      IMAGE_FILE_MACHINE_IA64 0x0200Intel Itanium
      IMAGE_FILE_MACHINE_AMD64 0x8664x64

    • Characteristics：最后一个属性成员占2字节，该成员是按bit位来看的，属性值也是多个属性的组合（或运算）。此处应该有图

  • 第三个成员是可选PE头(_IMAGE_OPTIONAL_HEADER)：

    • AddressOfEntryPoint：指出文件被执行时的入口地址，这是一个RVA地址，如果在一个可执行文件上附加了一段代码并想让这段代码首先被执行，那么只需要将这个入口地址指向附加的代码

    • ImageBase：内存镜像基址，指出文件的优先装入地址。也就是说当文件被执行时，如果可能的话，Windows优先将文件装入到由ImageBase字段指定的地址中。 一般EXE文件的默认为00400000H，而DLL文件的默认为10000000H

      关于重定向

      当这个地址被其他程序或者模块霸占时，系统会进行重定向，将它放置到其他地址处

      链接器产生可执行文件的时候对应这个地址来生成机器码，所以当文件被装入这个地址时不需要进行重定位操作，装入的速度最快。如果文件被装载到其他地址的话，将不得不进行重定位操作，这样就要慢一点。 对于EXE文件来说，由于每个文件总是使用独立的虚拟地址空间，优先装入地址不可能被其他模块占据，所以EXE总是能够按照这个地址装入。这也意味着EXE文件不再需要重定位信息。 对于DLL文件来说，由于多个DLL文件全部使用宿主EXE文件的地址空间，不能保证优先装入地址没有被其他的DLL使用，所以DLL文件中必须包含重定位信息以防万一。 因此，在前面介绍的 IMAGE_FILE_HEADER 结构的 Characteristics 字段中，DLL 文件对应的IMAGE_FILE_RELOCS_STRIPPED位总是为0，而EXE文件的这个标志位总是为1。 在链接的时候，可以通过对link.exe指定/base:address选项来自定义优先装入地址，如果不指定这个选项的话则为默认。

    • SectionAlignment和FileAlignment：内存对齐= 00001000H，即4096字节和文件对齐= 00000200H，即512字节。也就是说，每个节被装入的地址必定是本字段指定数值的整数倍。文件对齐和内存对齐的目的是提高效率

    • DataDirectory：16个元素的结构体数组。
